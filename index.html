Origines de Samba
Décembre 1991 - Janvier 1992 : Andrew Tridgell développe la première version de Samba comme étudiant en doctorat à l'Université nationale australienne
1992 : Premières releases (0.1, 0.5, 1.0), Tridgell l'appelle "Unix file server for DOS Pathworks"
Décembre 1993 : Sortie de nbserver 1.5, premier release avec support client et adoption de la GPL2
Janvier 1999 : Version 2.0.0, support authentication depuis Windows NT Primary Domain Controller
23 Septembre 2003 : Version 3.0.0, upgrade majeure avec capability de joindre Active Directory
2006+ : Samba 4 développé pour support complet comme Active Directory Domain Controller
2024-2025 : Samba 4.21, 4.22, 4.23 avec SMB over QUIC et monitoring Prometheus

Motivation Originelle

Problème initial : Andrew Tridgell, étudiant en doctorat à l'ANU, ne pouvait pas utiliser simultanément le client PC-NFS et DEC Pathworks sur le même système.

Solution créative : Utiliser un "packet sniffer" pour analyser le protocole SMB de DEC Pathworks et l'implémenter sur Unix.

Découverte inattendue : Le code interopérait avec Microsoft Windows (qui utilisait également SMB), créant ainsi une solution d'interopérabilité accidentelle.

Clarification importante : Samba est antérieur à Windows NT (Janvier 1992 vs Août 1993). Ce n'est pas un clone Windows NT, mais une implémentation du protocole SMB d'origine.

But et Objectifs de Samba
  • Partage de fichiers : Créer des partages accessibles depuis Windows, macOS et Linux de manière transparente
  • Authentification : Gérer les utilisateurs et groupes, intégration complète avec Active Directory
  • Partage d'imprimantes : Permettre aux clients d'imprimer via le serveur Samba (CUPS)
  • Contrôle d'accès : Gestion fine des permissions (ACL) et des droits utilisateur
  • Interopérabilité : Pallier les écarts propriétaires Microsoft - Unix/Linux
  • Solution gratuite : Offrir une alternative open-source sous GPL aux services Windows propriétaires

Architecture de Samba
Couche 1 : Protocoles Réseau SMB (Server Message Block) et CIFS (Common Internet File System) - TCP port 445, legacy NetBIOS port 139
Couche 2 : Démons Core smbd (services fichiers/imprimantes)
nmbd (résolution NetBIOS),
winbindd (authentification NT/AD)
Couche 3 : Gestion Utilisateurs SAM (Security Account Manager), LDAP, Active Directory complet (Samba 4), authentification Kerberos
Couche 4 : Partages & Ressources Partages de fichiers, support CUPS pour imprimantes, DFS (Distributed File System)

Composants Principaux
  • smbd : Démon principal gérant connexions SMB/CIFS, partages, authentification et opérations fichiers
  • nmbd : Gestion du protocole NetBIOS et résolution de noms (WINS - Windows Internet Name Service)
  • winbindd : Passerelle entre systèmes Unix/Linux et domaines NT/Active Directory. Reste essentiel pour accéder à AD, certains paramètres dépréciés depuis Samba 4.21+
  • samba-tool : Outil CLI pour gérer Samba 4 (provision AD, gestion domaines, utilisateurs)
  • testparm : Validation et analyse de la configuration smb.conf
  • net : Utilitaires réseau (gestion AD, groupes, partages, joins domaine)
  • pdbedit : Gestion de la base de données des utilisateurs locaux

Modes de Fonctionnement
1. Standalone Server ou Workgroup Mode (Mode Groupe de Travail) Pas de contrôle centralisé, authentification locale sur chaque machine. Configuration simple sans intégration au domaine
2. Domain Member (Membre de Domaine) Intégration dans un domaine Active Directory existant (Windows ou Samba). Authentification centralisée.
3. Active Directory DC (Contrôleur de Domaine AD) Samba 4 agit comme serveur AD complet (depuis v4.0), remplaçant entièrement les services Windows

Authentification : Identifier les Utilisateurs
Problème :

Comment vérifier qu'un utilisateur est vraiment "stephane" et pas quelqu'un d'autre ?

Solution 1 : NTLM / NTLMv2 (Ancienne méthode)

L'utilisateur envoie son mot de passe haché (pas le mot de passe en clair). Le serveur compare avec le hash stocké.

  • Rapide mais moins sûr
  • Utilisé par les vieux systèmes Windows (XP, Server 2003)
Solution 2 : Kerberos (Méthode moderne recommandée)
  • L'utilisateur demande un "ticket" au serveur avec son mot de passe
  • Le serveur génère un ticket sécurisé
  • L'utilisateur utilise ce ticket pour accéder à tous les services sans retaper son mot de passe (Single Sign-On)
  • Plus sûr et pratique
  • Standard des environnements professionnels (Active Directory)

Annuaires : Où Stocker les Utilisateurs ?
Problème :

Comment stocker centralisé tous les utilisateurs, groupes et ordinateurs du réseau ?

Solution 1 : LDAP (Lightweight Directory Access Protocol)

Protocole pour stocker et chercher utilisateurs, groupes, ordinateurs de manière centralisée. C'est l'annuaire téléphonique du réseau : on le consulte pour trouver qui est qui.

Solution 2 : Active Directory (AD)

Base de données complète basée sur LDAP qui gère :

  • Utilisateurs et groupes
  • Ordinateurs du domaine
  • Permissions et stratégies
  • Authentification centralisée

Samba 4 = Active Directory complet sur Linux (gratuit !)

Contrôle d'Accès : Qui Peut Faire Quoi ?
Problème :

Comment s'assurer que seules les bonnes personnes accèdent à certains fichiers ou services ?

Solution 1 : ACL (Access Control List)

Liste fine des permissions par utilisateur/groupe pour chaque fichier ou dossier.

Exemples :

  • "stephane peut lire le dossier IT"
  • "IT_Team peut modifier les fichiers"
  • "guests lecture seule"
Solution 2 : SMB3 Chiffrement + Signature

Les données en transit sont protégées :

  • Chiffrées → personne ne peut lire les données
  • Signées → personne ne peut modifier les données

Avantages & Limitations
✓ Avantages
  • Gratuit et open-source sous GPL - aucun coût de licence
  • Compatibilité excellente avec tous clients Windows (moderne + legacy)
  • Faible consommation de ressources CPU/RAM vs Windows Server
  • Très stable, fiable, production-ready depuis 30+ ans
  • Communauté active, support professionnel disponible (SerNet SAMBA+)
  • Peut remplacer entièrement Active Directory Microsoft
⚠ Limitations
  • Configuration complexe pour débutants (fichier smb.conf)
  • Courbe d'apprentissage importante (administration Linux requise)
  • Quelques fonctionnalités AD très avancées moins développées que Microsoft
  • Documentation parfois fragmentée/obsolète pour cas complexes
